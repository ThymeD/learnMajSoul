# Cursor 工作流固化

## 目标
让 AI 在**新窗口**或**新对话**里也能继续干活：所有状态（待办、需求清单、用户缺陷、修复进度）都从项目文件读取、并在每次迭代后被更新回项目里。

## 信息源（每次对话都要先读）
1. `collaboration.md`
   - 待办事项（生产问题/缺陷修复进度）
   - 最新任务（用户可复制给当前窗口）
2. `requirements/`
   - 需求设计文档（例如 `requirements/hand-view-requirements.md`）
   - 作为“实现后必须刷新”的需求真相源
3. `src/` 与 `tests/`
   - 用于验证实现是否满足需求与验收标准

## 处理新需求的规则（必须固化到项目）
当用户在聊天中提出一个**新的需求**（功能/交互/边界条件）时，AI 必须：
1. 在 `collaboration.md` 中把该需求加入“最新任务（用户可复制给当前窗口）/ 需求清单（用于恢复项目真相）”，并标注优先级与预期产出位置（例如更新哪个 `requirements/*.md`）。
2. 在对应的需求设计文档中新增/更新条目：
   - 更新对应的 `HAND-xxx` 或新增条目 ID
   - 补全验收标准/边界条件/交互说明
3. 实现后必须把结果同步回需求设计文档（“已完成/已验证/未验证”）。

## 处理用户缺陷的规则（必须跟踪闭环）
当用户提出一个**用户缺陷**（表现不符合预期）时，AI 必须：
1. 在 `collaboration.md` 增加一条缺陷记录（复现步骤、期望、实际、初步根因/猜测）。
2. 代码修复后补全：
   - 影响范围
   - 验证方式与证据（例如 `npm run test:run` 通过、Playwright 用例通过等）
   - 修复进展状态（进行中/已修复/已验证/关闭）
3. 只有在用户确认“已满足预期”后，才能把缺陷标记为关闭。

## 开发与验证建议
- 每次完成一项实现：优先运行 `npm run test:run`，并把验证结果写入 `collaboration.md` 对应条目。
- 若 UI 交互涉及 DnD/开关/联动：补充 Playwright/手动验证步骤到缺陷记录或需求验收说明。
