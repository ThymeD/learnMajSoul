# Cursor 工作流固化

## 目标
让 AI 在**新窗口**或**新对话**里也能继续干活：所有状态（待办、需求清单、用户缺陷、修复进度）都从项目文件读取、并在每次迭代后被更新回项目里。

## 信息源（每次对话都要先读）
1. `collaboration.md`
   - 最新任务（用户可复制给当前窗口）
   - 需求清单（用于恢复项目真相）
   - 缺陷记录与修复状态
2. `requirements/INDEX.md`
   - 全项目统一需求索引（模块、文档路径、状态）
3. 具体需求文档（如 `requirements/hand-view-requirements.md`）
4. `src/` 与 `tests/`
   - 用于验证实现是否满足需求与验收标准

## 处理新需求的规则（必须固化到项目）
当用户提出新需求时，AI 必须：
1. 先在 `collaboration.md` 登记需求任务（含优先级、影响范围、预期产出位置）。
2. 在 `requirements/INDEX.md` 登记模块需求状态（新增/变更中/已实现/已验证）。
3. 在对应需求文档中新增或更新条目（验收标准、边界条件、交互说明）。
4. 实现完成后刷新需求文档状态与验证结果，不允许“只有代码没有需求设计”。

## 需求冲突处理（必须二次确认）
当新需求与现有需求冲突时，AI 必须：
1. 明确冲突点（冲突条目、行为差异、影响范围）。
2. 输出变更任务清单（文档更新、代码调整、测试补充、回归验证）。
3. 请求用户确认处理方向后再实施代码改动。

## 方案对比决策（有更优实践时必须给选项）
当存在更优方案时，AI 必须给出至少 2 个方案并比较：
- 实现复杂度
- 迭代风险
- 对现有功能影响
- 推荐结论
由用户决策后执行。

## 处理用户缺陷的规则（必须跟踪闭环）
当用户提出缺陷时，AI 必须：
1. 在 `collaboration.md` 记录缺陷（期望/实际、复现步骤、影响范围、状态）。
2. 修复后补全验证证据（自动化测试/手动验证/关键截图说明）与修复提交。
3. 仅在用户确认后将缺陷状态改为“已关闭”。

## 根因分类与防重复犯错
每条缺陷必须标注根因类型：
- `REQ_CHANGE`：用户需求调整
- `REQ_GAP`：需求文档不清或缺失
- `AI_IMPL`：AI实现问题
- `TEST_GAP`：测试覆盖不足

同一模块同一根因重复 >=2 次时，必须新增防重复动作并写回项目文件：
- 增补验收清单
- 增加测试用例/回归步骤
- 增加实现前检查项

## 开发与验证建议
- 每次完成实现后，优先运行 `npm run test:run`，并把结果回填到 `collaboration.md`。
- 涉及复杂交互（DnD、联动开关、跨区同步）时，补充 Playwright 或明确手动回归步骤。

## 上下文与 token 控制
- 进入任务前先读取：`collaboration.md`、`requirements/INDEX.md`、对应模块需求文档。
- 避免整文件读取超大代码文件，优先按符号或区段读取。
- 若任务跨越过多文件，先拆分子任务并分阶段交付，避免单次上下文过载。
- 参考：`.cursor/rules/context-efficiency.md`
