# 雀魂麻将游戏攻略 - 项目全局规则

## 项目概述

本项目旨在实现雀魂麻将游戏攻略系统，采用模块化开发，每个功能可独立验证，最终整合为完整系统。

## 开发策略

- **小步快跑**：分模块实现，每个功能打磨完成后独立验证
- **先设计后实现**：AI 思考清楚方案后，先提交用户确认
- **方案需包含预期效果**：让用户明确知道完成后能达到什么效果
- **草稿区开发**：未指定在已有模块优化的新需求，默认先在草稿区实现，验证通过后迁移到对应功能页面，草稿区代码需与已有功能隔离
  - 验证新功能时，直接覆盖现有草稿区内容（覆盖前主动找用户确认）
  - 旧内容做好备份和命名（如 `DraftView_backup_日期.vue`）

## Git 管理规范

- 每次独立特性完成后**必须提交到本地**
- 提交信息清晰描述完成内容
- 如需大改动，先从主分支拉取备份分支

## 技术规范

- **UI 组件库**：优先使用 Element Plus，组件缺失时使用自定义组件但保持风格一致
- **文档参考**：遇到 AI 训练语料可能过时时，以官网 https://element-plus.org/zh-CN/ 为准
- **TypeScript**：全项目使用 TypeScript，严格类型检查

## 验证标准

每个模块完成后需满足：

1. 功能可独立运行验证
2. 测试用例通过
3. 代码符合项目规范
4. 构建通过（npm run build）

## 雀魂麻将的基础胡牌数量规则

- 正常胡牌14张牌（13张手牌+1张听牌）
- 开杠比正常胡牌多一张，每个杠多一张（杠=4张相同的牌）

## 需求与缺陷处理流程（强制）

> ⚠️ **核心规则**：每次需求/缺陷都必须经过"分析→计划→固化→确认"才能进入编码

### 触发条件

当用户提出以下任何形式的表述时，**自动触发**本流程：

- "新增..."、"添加..."、"实现..."、"做一个..."
- "修复..."、"解决..."、"排查..."、"修bug"
- "优化..."、"改进..."、"重构..."
- 任何功能、缺陷、优化相关的需求描述

> 注意：**通用问答**（如"这个代码什么意思"、"帮我解释一下"）不触发本流程

---

### 处理流程（强制顺序）

#### 1. 分析阶段（自动）

**操作**：

- 理解用户需求的本质目标
- 分析现有代码结构、相关模块依赖
- 识别需要改动的影响范围
- 评估任务改动级别

**产出**：1-3句话简要分析说明

---

#### 2. 计划阶段（自动）

**操作**：

- 将需求拆解为具体可执行的任务
- 为每个任务分配改动级别
- 指定负责的子 Agent
- 预估工作量

**产出**：TODO.md 任务清单格式

---

#### 3. 持久化阶段（自动）

**操作**：

- 立即更新 TODO.md，添加新任务清单
- 标注所有任务状态为 `⏳ pending`
- 更新 CURRENT.md 当前会话状态

**文件位置**：

- `TODO.md` - 项目根目录
- `.opencode/session/CURRENT.md` - 会话状态

---

#### 4. 确认阶段（等待）

**操作**：

- 展示完整的任务计划
- 等待用户明确确认后才能开始执行

**确认格式**：

```
📋 任务计划已生成：

| 状态 | 任务 | 改动级别 | 负责Agent |
|------|------|----------|----------|
| ⏳ pending | 任务1：xxx | 小改动 | coder |
| ⏳ pending | 任务2：xxx | 中改动 | tester |

请确认后我开始执行。
```

---

### 改动级别定级标准

**判断维度1：改动范围**
| 改动范围 | 级别 |
|----------|------|
| 单文件 | 微调 |
| 2-3个文件 | 小改动 |
| 4-10个文件 | 中改动 |
| 10+文件/跨模块 | 大需求 |

**判断维度2：改动类型**
| 改动类型 | 级别提升 |
|----------|----------|
| 错别字/样式调整 | 维持微调 |
| 业务逻辑修改 | 至少小改动 |
| 新增功能 | 至少中改动 |
| 新增页面/模块 | 大需求 |

**判断维度3：影响范围**
| 影响范围 | 级别提升 |
|----------|----------|
| 局部 | 维持 |
| 单模块 | +1级 |
| 多模块/全局 | +2级 |

---

### 改动级别与流程

| 级别   | 涉及环节 | 流程                           |
| ------ | -------- | ------------------------------ |
| 微调   | 1环节    | coder → tester                 |
| 小改动 | 2环节    | coder → code-reviewer → tester |
| 中改动 | 3环节    | 完整流程（含技术方案/UI设计）  |
| 大需求 | 4环节    | 完整流程 + 需求规格化          |


## POC 验证机制

### 定位

POC 是**辅助验证工具**，用于在正式交付流程之前验证理解或技术方案，**不是项目交付流程的一部分**。

### 触发场景

| 场景       | 说明                         | 处理方式             |
| ---------- | ---------------------------- | -------------------- |
| 需求不清晰 | 用户描述模糊，需要先验证理解 | POC 验证后转正式流程 |
| 技术验证   | 不确定技术方案是否可行       | POC 验证后转正式流程 |
| 界面验证   | 不确定界面是否符合预期       | POC 验证后转正式流程 |
| 改动风险高 | 大量改动已有代码，担心影响   | POC 验证后转正式流程 |

### POC 目录结构

```
project/
├── .poc/                    # POC 独立目录
│   ├── scenario-1/         # 场景1
│   │   ├── SPEC.md        # POC 方案描述
│   │   ├── demo/          # 演示代码
│   │   └── README.md      # 说明
│   └── scenario-2/        # 场景2
├── .opencode/              # 项目交付
├── src/                    # 项目代码
└── ...
```

### POC 工作流程

```
用户/主 agent 判断需要 POC
    ↓
创建 .poc/scenario-N/
    ↓
调用相关 子 agent 出方案
    ↓
展示方案供用户选择
    ↓
用户确认方案
    ↓
验证（技术/界面/功能）
    ↓
用户确认符合预期 → 合并到项目 → 转正式交付流程
            或
         放弃 → 删除 .poc/
```

### POC vs 项目交付

| 维度     | POC                   | 项目交付       |
| -------- | --------------------- | -------------- |
| **触发** | 需求不清晰/技术不确定 | 需求明确       |
| **目的** | 验证理解              | 正式交付       |
| **代码** | 独立，不影响项目      | 合并到正式代码 |
| **状态** | 临时，可放弃          | 持久化         |
| **流程** | POC验证 → 合并/放弃   | 需求处理流程   |

## 用户确认节点

以下阶段完成后**必须**等待用户确认：

| 阶段       | 确认内容           | 确认方式             |
| ---------- | ------------------ | -------------------- |
| 任务计划   | TODO.md 任务清单   | 必须确认后才开始执行 |
| 需求规格化 | SPEC.md 需求清单   | 大需求 必须确认      |
| 技术方案   | 技术选型、架构设计 | 大需求 必须确认      |
| UI设计     | 设计稿、原型       | 需要 UI 时必须确认   |
| 功能验收   | 功能测试结果       | 每个模块完成后确认   |

确认方式：AI 输出确认提示，用户回复"确认"或提出修改意见。

## 状态持久化

### 保存目录结构

```
project/
├── .opencode/               # opencode 工作目录
│   └── session/
│       ├── CURRENT.md      # 当前工作状态
│       └── HISTORY.md      # 历史记录
├── TODO.md                 # 项目任务清单
```

### 触发时机

| 时机           | 保存内容                           |
| -------------- | ---------------------------------- |
| **新需求接收** | 更新 TODO.md 任务清单 + CURRENT.md |
| 每个任务完成   | 更新 TODO.md 任务状态为 ✅ done    |
| 每完成一个阶段 | 更新 CURRENT.md 当前状态           |
| 窗口启动       | 读取 CURRENT.md + TODO.md 恢复状态 |

### 启动恢复机制（强制自动）

**新窗口启动时**：

1. 自动读取 CURRENT.md 了解当前工作状态
2. 自动读取 TODO.md 了解待办任务
3. 自动询问用户是否继续当前任务

> ⚠️ 这是自动行为，**不需要用户提示才执行**

### 原则

- 少量对话丢失可接受
- 任务进度都在 TODO.md 中（git 可追溯）
- 新窗口可以读取 CURRENT.md 恢复上下文

## 项目交付任务执行流程

### 概述

在"需求与缺陷处理流程"确认任务计划后，按以下步骤执行每个任务。

---

### 执行阶段

**操作**：

- 按 TODO.md 中改动级别顺序执行
- 每完成一个任务立即更新 TODO.md 状态为 ✅ done
- 使用子 Agent 协同（根据任务类型）
- 遇到阻塞时记录到 CURRENT.md

---

### 验证阶段

**操作**：

- 运行测试 `npm run test:run`
- 运行构建 `npm run build`
- 验证功能正常

**验收标准**：

- 测试全部通过
- 构建成功
- 功能符合预期

---

### 提交阶段

**操作**：

- 分批提交到本地仓库
- 提交信息清晰描述完成内容
- 更新 CURRENT.md 当前状态

---

### 流程图

```
确认任务计划
    ↓
执行任务（按改动级别）
    ↓
实时更新 TODO.md 状态
    ↓
验证（测试+构建）
    ↓
提交到本地仓库
    ↓
更新 CURRENT.md
```

---

```markdown

> 更新日期：YYYY-MM-DD
```

```markdown
- [ ] 用户确认任务计划
```
